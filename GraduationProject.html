Mobile Multiplayer Real Time Strategy Game

For this project I choose to go deeper into networking. Where I used a framework for the racing game (The Mobile Multiplayer Race Game), I now wanted to write the network code myself. The goal that I had set myself was to make a multiplayer game with a dedicated server. That means my server validates everything the players do. Only when my server approves an action is it synchronized with the players.

Regarding the game, I choose to make it a mobile game. This because I already had some experience with mobile games and because it is more accessible. I also opted to make it a Real Time Strategy (RTS) game. In this way I hoped to encounter various multiplayer facets. I learned a lot about how to build a custom .Net server from scratch. How do I put the control of the game at the server and how the server synchronizes all permissible actions.

Finally, regarding the different technical areas, I have linked a database to my server. This ensures that I can save player data and that it is preserved between different game sessions. Think of a match history and match stats.

Server


public static void Start(int maxPlayers, int port){
    Server.maxPlayers = maxPlayers;
    Server.port = port;

    Console.WriteLine("Starting server");
    InitServerData();

    tcpListener = new TcpListener(IPAddress.Any, Server.port);
    tcpListener.Start();
    tcpListener.BeginAcceptTcpClient(new AsyncCallback(TcpConnectCallback), null);

    udpListener = new UdpClient(port);
    udpListener.BeginReceive(UdpReceiveCallback, null);

    Console.WriteLine($"Server Started on {Server.port}.");
}

For this project I implemented a UDP (User Datagram Protocol) and TCP (Transmission Control Protocol) connection. Mainly I have used TCP because it is a bit more reliable. UDP is better to use when I can miss one data cycle for example. So when I established a connection I am able to send and receive data from the server. So when the player opens the app the game tries to connect to the server. Ones connected the server is gone check if its a new player or not. If so the player needs to fill in a username. The server then saves the username and player in the database and the player is ready to start a match. Because we know which player connected to the server we are able to save some player data in the database (see images).

When the player wants to start a match, he first sends a packet to the server asking if he can play. If the server allows the player to start a match, the server puts the player in to a lobby to wait for an opponent. Ones there is another player that willing to play a match. The server checks if there is a lobby available to join and adds the player to it, If not the server creates a new lobby. When both players are ready. The server sends a message to both players with the map info. The info tells the players where the player can find certain resources and where each players base building is placed on the map. If both players fully loaded the map the server tels both players to start the countdown. Ones its zero the match starts. Both players are now able to place buildings, create troops and so on.

With an RTS game like this you usually move a lot of troops at the same time. To make it a bit easier on mobile I implemented some kind of grouping mechanic. So when a unit is spawned and it spawns near a the same unit it automaticly group of it. When the player tries to move a unit that's in a group, it automatically moves the whole group. This makes it easier to move a lot of troops/units at the same time. Attacking works kinde simulair. If one unit in a group starts to attack the whole groups starts to attack. So for example if one unit in the group is getting in range of an enemy, it tells the whole group that there is an enemy in range and we should attack.

There are like 10 different units all with different characteristics like:

  • Cost
  • Health
  • Range
  • Damage
  • Hit Speed
  • Move Speed
  • Size Radius
  • Target Type

These characteristics give me the opportunity create a lot of different units. For example i've got units that can only attack buildings. Units can be ranged or melee, air or ground, i've got buildings and troops and so on.

But most importantly everything is validated on the server. If the player wants to perform an action. The player sends a packet to the server with the action he wants to perform and if necessary some additional information. The server checks if this is possible. If this is the case the server sends the action to all the players in the match. Each player then performs this action.
